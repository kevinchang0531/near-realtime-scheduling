CHALLENGE

問題本質：動態時窗資源指派最佳化

時間窗約束

衛星對目標的可拍窗、對地面站的可通聯窗皆為短暫且固定的天文事件。

資源衝突

單衛星同時段只能做一件事；天線、頻段、儲存、電力、姿態機動時間皆有限。

不確定性

緊急訂單插入、雲量/天氣變化、地面站故障、衛星異常 → 必須滾動重排。

⚡ 「近即時」的明確定義

緊急訂單自接收到產出可執行排程方案 ≤ 5 分鐘（目標 ≤ 2 分鐘），透過事件驅動的增量重排達成。

REQUIREMENTS

系統需求 — 功能 (FR) 與非功能 (NFR)

功能需求 FR

  • FR-01 訂單接收／優先級分類（一般·緊急·VIP）
  • FR-02 星曆維護與可見窗計算（TLE/SGP4）
  • FR-03 地面站資源狀態維護
  • FR-04 初始排程產生與可視化
  • FR-05 事件觸發式動態增量重排
  • FR-06 衝突偵測與優先權搶占
  • FR-07 衛星指令 + 地面站 Pass Plan 產出
  • FR-08 執行監控、達成率與計費依據
  • FR-09 What-if 模擬與人工覆核

非功能需求 NFR

  • 效能：緊急重排 P95 ≤ 5min；全域排程 ≤ 60s
  • 可用性：核心服務 99.9%，HA 無單點故障
  • 可擴充：個位數→數百顆衛星不需重構
  • 可追溯：每筆排入/排除/搶占決策可審計
  • 安全：分級管控 · mTLS · RBAC · 操作日誌
  • 時間同步：全系統 UTC，NTP/PTP
納入率 ≥ 95% 利用率 +20% 衝突 = 0

OPERATIONAL SCENARIOS

操作情境

A · 一般訂單（批次）

白天下單 → 週期性彙整 → 全域最佳化 → 隔日任務計畫下達。

B · 緊急訂單插入 核心

災害/突發事件 → 30s 算可見窗 → 增量重排 → 衝突則優先權搶占 → 人工 1-click 覆核 → 下達 → 通知被搶占客戶。

C · 地面站資源異動

天線故障/天氣惡化 → 受影響下行任務自動改派他站或順延至下一通聯窗。

D · 衛星異常 / 新 TLE

新星曆進入 → 重算可見窗 → 受影響任務局部重排。

E · What-if 人工介入

模擬接受 VIP 訂單對達成率/違約金的影響後核准。

SCHEDULING MODEL · CORE

排程模型 — 三層混合式策略

分鐘級 · 離線

① 批次全域最佳化

MILP (CP-SAT/Gurobi) 求小規模最優 · 或 ALNS / 遺傳演算法 / 蟻群求大規模近優 → 產出高品質基準排程。

秒級 · 事件觸發

② 動態增量重排

以基準解為起點，Rolling Horizon 滾動時域 + 修復式插入啟發，僅在受影響時間窗局部重算 → 達成「近即時」。

毫秒~秒 · 規則

③ 優先權搶占與衝突解決

以「優先權 + 違約成本」貪婪/規則決策，可解釋、可審計；高風險者觸發人工覆核 (Human-in-the-loop)。

目標函數：max Σ(訂單收益 × 優先權) − 機動成本 − SLA 違約金 − 資源懲罰　｜　進階：RL / 隨機規劃 學習為未來緊急訂單保留資源

MODEL RATIONALE

為何這樣選模型？

需求對應模型理由
高品質 · 大規模ALNS / GA / 蟻群數百衛星×數千訂單下 MILP 無法在時限內求解，元啟發式秒~分鐘給近優解
嚴格最優 · 小規模MILP / CP-SAT訂單少時求全域最優，作為基準與驗證
近即時動態調整Rolling Horizon + 修復啟發只重排受影響局部，避免全域重算，滿足分鐘級 SLA
緊急搶占優先權規則 + 成本決策快速、可解釋、可審計
不確定性 / 未來彈性強化學習 RL / 隨機規劃學習「保留資源給未來緊急訂單」策略（Phase 2 增強）

務實建議：MVP 先用 OR-Tools CP-SAT 做批次層 + 貪婪插入/ALNS 做增量層，成本低、可快速驗證；規模上來再引入商用求解器或 RL。
